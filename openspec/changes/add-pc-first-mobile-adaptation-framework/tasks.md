## 1. Contract

- [ ] 1.1 为 `GameManifestEntry` 增加显式移动端支持字段与壳目标字段，并定义默认值与校验规则。
- [ ] 1.2 更新清单生成与消费链路，让运行时、游戏详情页和后续分发逻辑都能读取这些字段。
- [ ] 1.3 补充文档，规定新增游戏时必须先声明自己的移动支持 profile。

## 2. Framework

- [ ] 2.1 在框架层实现 `landscape-adapted` 适配壳：安全区、横屏视口、可折叠侧栏、压缩 HUD、触控尺寸默认值。
- [ ] 2.2 为 hover / 精细鼠标交互提供统一移动端替代方案：点击切换、长按预览、底部抽屉/模态面板。
- [ ] 2.3 将当前横屏提示和缩放兜底改为 manifest 驱动，而不是全局硬编码路由策略。
- [ ] 2.4 定义移动端视觉与性能降级策略，避免动画和特效在手机上成为瓶颈。

## 3. DiceThrone Pilot

- [ ] 3.1 审查王权骰铸当前桌面 UI 中依赖 hover、细粒度拖拽、固定宽高和侧栏常驻的部分。
- [ ] 3.2 用新的 `landscape-adapted` 框架能力改造王权骰铸，保留桌面布局为主版本，不新增一套独立移动端 Board。
- [ ] 3.3 在王权骰铸 manifest 中声明移动支持 profile、推荐方向和允许的壳目标。
- [ ] 3.4 补充横屏手机与平板验证，包括手牌、交互面板、状态查看、关键操作路径。

## 4. Container Strategy

- [ ] 4.1 明确并文档化 H5 / PWA 为唯一真实运行时，App WebView 为后续分发壳。
- [ ] 4.2 明确并文档化微信小程序 `web-view` 的限制与定位，仅作为未来入口容器，不作为本次主运行时。
- [ ] 4.3 定义哪些游戏可以被标记为适合进入 App WebView / 小程序 `web-view`，避免所有游戏默认暴露到壳里。

## 5. Developer Enablement

- [ ] 5.1 从王权骰铸试点中提炼“游戏层接入移动适配框架”的标准步骤、检查项和常见例外。
