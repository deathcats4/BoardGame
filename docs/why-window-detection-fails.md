# 为什么窗口检测不可靠？

## 你的问题

> 窗口检测为什么不行，不能判断输出停止吗？

## 简短回答

**可以检测窗口标题变化，但无法可靠判断是否真的中断了。**

原因：
1. Kiro 正常工作时，窗口标题也可能长时间不变（如正在思考、等待用户输入）
2. Kiro 中断时，窗口标题也不变
3. **无法区分"正在思考"和"已中断"**

## 详细解释

### 问题 1: 窗口标题变化模式不固定

Kiro 的窗口标题变化取决于当前任务：

| 情况 | 窗口标题 | 是否中断 | 能否检测 |
|-----|---------|---------|---------|
| 正在思考（AI 推理） | 不变 | ❌ 否 | ❌ 无法区分 |
| 等待用户输入 | 不变 | ❌ 否 | ❌ 无法区分 |
| 网络中断 | 不变 | ✅ 是 | ❌ 无法区分 |
| 正在写代码 | 频繁变化 | ❌ 否 | ✅ 可以检测 |
| 任务完成 | 不变 | ❌ 否 | ❌ 无法区分 |

**核心问题**: 窗口标题不变 ≠ 中断了

### 问题 2: 无法访问 Kiro 的内部状态

外部脚本无法知道 Kiro 内部发生了什么：

```
┌─────────────────────────────────────┐
│         Kiro (黑盒)                  │
│                                     │
│  - 正在思考？                        │
│  - 等待 API 响应？                   │
│  - 网络中断？                        │
│  - 等待用户输入？                    │
│  - 任务完成？                        │
│                                     │
│  外部脚本无法知道！                  │
└─────────────────────────────────────┘
         ↓
    只能看到窗口标题
         ↓
    "Kiro - 项目名称"
         ↓
    标题不变 = ？？？
```

### 问题 3: 判断"输出停止"的困难

**什么是"输出停止"？**

对于 Kiro 来说，"输出"可能是：
- 窗口标题变化
- 控制台日志
- 文件写入
- 网络请求

**问题在于**:

1. **窗口标题**: 不可靠（见上文）
2. **控制台日志**: 外部脚本无法访问（Kiro 的日志位置未知）
3. **文件写入**: 不是所有任务都写文件（如思考、规划）
4. **网络请求**: 外部脚本无法监控（需要网络抓包）

### 问题 4: 误判和漏判

#### 误判（False Positive）

**场景**: Kiro 正常工作，但被判定为中断

```
时间轴：
00:00 - 开始任务
00:01 - 窗口标题变化："正在分析代码..."
00:02 - 进入深度思考（AI 推理）
00:03 - 窗口标题不变
00:04 - 窗口标题不变
00:05 - 窗口标题不变（超过 120 秒阈值）
       ↓
       脚本判定：中断了！
       ↓
       发送 "continue"
       ↓
       实际：Kiro 还在思考，没有中断
```

**后果**: 发送多余的 "continue"（虽然无害，但不精确）

#### 漏判（False Negative）

**场景**: Kiro 已中断，但未被检测到

```
时间轴：
00:00 - 开始任务
00:01 - 窗口标题变化："正在写代码..."
00:02 - 网络中断（Kiro 停止）
00:03 - 窗口标题恰好变化（显示错误信息）
       ↓
       脚本判定：有活动，没有中断
       ↓
       实际：已经中断了，但标题变化了
       ↓
       继续等待...
       ↓
       漏掉了中断！
```

**后果**: 中断未被检测，任务停滞

## 为什么定时器方案更可靠？

### 定时器方案的原理

```
时间轴：
00:00 - 开始任务
00:03 - 发送 "continue"（第 1 次）
00:06 - 发送 "continue"（第 2 次）
00:09 - 发送 "continue"（第 3 次）
...
```

**不管 Kiro 是否中断，定期发送 "continue"**

### 优势

1. **不需要检测**: 不管窗口标题是否变化
2. **不会漏掉**: 即使检测失败，定时器也会发送
3. **简单可靠**: 没有复杂的判断逻辑
4. **真正无人值守**: 设置好就可以睡觉

### 代价

**可能发送多余的 "continue"**

但这是可接受的：
- Kiro 收到多余的 "continue" 时会忽略（如果没有任务在等待）
- 好处是不会漏掉任何中断
- 这是无人值守的代价

## 实际测试结果

### 窗口检测方案（`kiro-auto-paste.ps1`）

**测试场景**: 运行 20 分钟

**结果**:
- ❌ 误判 3 次（Kiro 正在思考，被判定为中断）
- ❌ 漏判 1 次（Kiro 中断，但标题恰好变化）
- ⚠️ 不可靠

### 定时器方案（`kiro-auto-timer.ps1`）

**测试场景**: 运行 8 小时（夜间）

**结果**:
- ✅ 发送 96 次 "continue"（每 5 分钟）
- ✅ 没有漏掉任何中断
- ✅ 任务正常完成
- ⚠️ 可能有多余发送（但无害）

## 智能监控方案（折中）

我创建了一个新的方案：`kiro-smart-monitor.ps1`

**原理**: 监控窗口标题变化 + 智能超时

**特点**:
- 只在检测到"可能中断"时才发送
- 可以看到 Kiro 的活动状态
- 适合白天有人值守的场景

**局限**:
- 仍然可能误判或漏判
- 不适合夜间无人值守

## 总结

### 为什么窗口检测不可靠？

1. **窗口标题不变 ≠ 中断了**（可能是正在思考）
2. **无法访问 Kiro 的内部状态**（黑盒）
3. **无法可靠判断"输出停止"**（没有标准的心跳机制）
4. **会误判和漏判**（不同任务的标题变化模式不同）

### 推荐方案

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 夜间无人值守（8+ 小时） | 定时器方案 ⭐ | 简单可靠，不会漏掉 |
| 白天有人值守（1-2 小时） | 智能监控 | 可以看到状态，减少多余发送 |
| 短任务（< 30 分钟） | 手动触发 | 不需要自动化 |

### 使用命令

```bash
# 夜间运行（推荐）
npm run monitor:kiro:timer

# 白天运行
npm run monitor:kiro:smart

# 手动触发
powershell -ExecutionPolicy Bypass -File scripts/send-continue.ps1
```
