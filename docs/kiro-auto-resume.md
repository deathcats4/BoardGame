# Kiro 自动恢复系统

## 概述

Kiro 自动恢复系统用于监控 Kiro 的执行状态，当检测到网络中断或任务停滞时，自动创建恢复标记触发 Hook 继续任务。

## 功能特性

- ✅ 自动监控日志文件活动
- ✅ 检测 Spec 执行状态
- ✅ 识别网络中断和任务停滞
- ✅ 支持多窗口/多会话
- ✅ 自动创建恢复标记
- ✅ 配合 Hook 自动恢复任务
- ✅ 调试模式（详细日志）
- ✅ 自动同意模式（跳过命令确认）
- ✅ 重试次数限制（防止无限循环）
- ✅ 智能计数重置（任务正常完成后自动重置）

## 快速开始

### 1. 启动监控

```bash
# 基础模式（默认配置）
npm run monitor:kiro

# 调试模式（显示详细日志）
npm run monitor:kiro:debug

# 自动同意模式（跳过命令确认）
npm run monitor:kiro:auto

# 完整模式（调试 + 自动同意）
npm run monitor:kiro:full

# 限制重试次数（自动同意 + 最多重试 3 次）
npm run monitor:kiro:retry3

# 限制重试次数（自动同意 + 最多重试 5 次）
npm run monitor:kiro:retry5

# 限制重试次数（自动同意 + 最多重试 20 次）
npm run monitor:kiro:retry20

# 自定义配置
node scripts/kiro-auto-resume.mjs --interval=20 --timeout=90 --max-retries=3

# 与开发服务器一起启动
npm run dev:with-monitor
```

### 2. 配置参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `--interval` | 检查间隔（秒） | 30 |
| `--timeout` | 无活动超时时间（秒） | 120 |
| `--debug` | 启用调试日志 | false |
| `--auto-approve` | 自动同意所有命令（跳过确认） | false |
| `--max-retries` | 最大自动恢复次数（0 = 无限制） | 0 |
| `--help` | 显示帮助信息 | - |

### 3. Hook 配置

脚本会自动创建 `.kiro/hooks/auto-resume.json`，内容如下：

```json
{
  "name": "Auto Resume on Interruption",
  "version": "1.0.0",
  "description": "检测到恢复标记时自动继续中断的任务（支持多窗口）",
  "when": {
    "type": "fileCreated",
    "patterns": [".kiro-resume-*"]
  },
  "then": {
    "type": "askAgent",
    "prompt": "检测到任务中断，请继续执行之前未完成的 Spec 任务。如果不确定上下文，请先检查最近的日志和 Spec 状态。"
  }
}
```

## 工作原理

### 监控流程

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 扫描日志目录，发现所有会话                                │
│    logs/app-2026-03-03.log → 会话 "2026-03-03"              │
│    logs/app-session-123.log → 会话 "session-123"            │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ 2. 为每个会话创建 SessionMonitor                             │
│    - 跟踪日志文件大小变化                                     │
│    - 检测 Spec 执行状态                                       │
│    - 记录最后活动时间                                         │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ 3. 定期检查（默认 30 秒）                                     │
│    - 日志文件是否有新内容？                                   │
│    - Spec 是否正在执行？                                      │
│    - 是否超过无活动超时时间？                                 │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ 4. 检测到中断时                                               │
│    - 创建恢复标记文件 .kiro-resume-<sessionId>               │
│    - 标记文件包含时间戳、会话 ID、中断原因                    │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ 5. Hook 自动触发                                              │
│    - 检测到 .kiro-resume-* 文件创建                          │
│    - 向 Kiro 发送"继续任务"指令                              │
│    - Kiro 恢复执行中断的 Spec                                │
└─────────────────────────────────────────────────────────────┘
```

### 多窗口支持

脚本通过日志文件名识别不同的会话：

- `logs/app-2026-03-03.log` → 会话 ID: `2026-03-03`
- `logs/app-session-abc.log` → 会话 ID: `session-abc`

每个会话独立监控，互不干扰。

## 使用场景

### 场景 1：长时间运行的 Spec

```bash
# 终端 1：启动监控
npm run monitor:kiro

# 终端 2：启动开发服务器
npm run dev

# 在 Kiro 中执行长时间 Spec
# 如果网络中断，监控脚本会自动检测并恢复
```

### 场景 2：多窗口开发

```bash
# 启动监控（支持多个 Kiro 窗口）
npm run monitor:kiro

# 窗口 1：开发游戏 A
# 窗口 2：开发游戏 B
# 窗口 3：修复 bug

# 每个窗口的任务独立监控和恢复
```

### 场景 3：调试模式

```bash
# 启用调试日志，查看详细监控信息
npm run monitor:kiro:debug

# 输出示例：
# 🐛 [2026-03-03] 检测到活动 (大小: 12345 bytes)
# 🐛 [2026-03-03] Spec 状态变化: 执行中
# ⚠️  [2026-03-03] 检测到执行中断（无活动 125秒）
# 🔄 [2026-03-03] 已创建恢复标记: .kiro-resume-2026-03-03
```

### 场景 4：自动同意模式（无人值守）

```bash
# 启用自动同意，Kiro 恢复后自动执行所有命令
npm run monitor:kiro:auto

# 适用场景：
# - 夜间运行长时间任务
# - 批量处理多个 Spec
# - CI/CD 环境
# - 信任的自动化流程

# ⚠️ 注意：自动同意会跳过所有确认，请确保任务安全可靠
```

### 场景 5：完整监控（调试 + 自动同意）

```bash
# 同时启用调试日志和自动同意
npm run monitor:kiro:full

# 适用场景：
# - 排查自动化流程问题
# - 验证恢复机制是否正常工作
# - 开发和测试监控脚本本身
```

### 场景 6：限制重试次数（防止无限循环）

```bash
# 最多自动恢复 3 次
npm run monitor:kiro:retry3

# 最多自动恢复 5 次
npm run monitor:kiro:retry5

# 自定义重试次数
node scripts/kiro-auto-resume.mjs --auto-approve --max-retries=10

# 适用场景：
# - 防止任务陷入无限重试循环
# - 网络不稳定时限制恢复次数
# - 需要人工介入的复杂任务
# - 避免资源浪费

# 工作原理：
# 1. 第 1 次中断 → 自动恢复（计数: 1/3）
# 2. 第 2 次中断 → 自动恢复（计数: 2/3）
# 3. 第 3 次中断 → 自动恢复（计数: 3/3）
# 4. 第 4 次中断 → 停止恢复，等待人工处理
# 5. 任务正常完成后，计数自动重置为 0
```

## 故障排查

### 问题 1：监控脚本无法启动

**症状**：运行 `npm run monitor:kiro` 报错

**解决方案**：
```bash
# 检查 Node.js 版本（需要 >= 16）
node --version

# 检查脚本权限
ls -l scripts/kiro-auto-resume.mjs

# 手动运行脚本查看详细错误
node scripts/kiro-auto-resume.mjs --debug
```

### 问题 2：Hook 未触发

**症状**：创建了恢复标记文件，但 Kiro 没有响应

**解决方案**：
```bash
# 1. 检查 Hook 配置是否存在
cat .kiro/hooks/auto-resume.json

# 2. 检查 Hook 是否启用
# 在 Kiro 中打开命令面板 → "Open Kiro Hook UI"

# 3. 手动测试 Hook
echo '{"test": true}' > .kiro-resume-test
# 观察 Kiro 是否响应

# 4. 清理测试文件
rm .kiro-resume-test
```

### 问题 3：误报（频繁触发恢复）

**症状**：任务正常执行，但监控脚本频繁创建恢复标记

**解决方案**：
```bash
# 增加超时阈值（默认 120 秒）
node scripts/kiro-auto-resume.mjs --timeout=300

# 或者增加检查间隔（默认 30 秒）
node scripts/kiro-auto-resume.mjs --interval=60
```

### 问题 4：日志文件不存在

**症状**：监控脚本提示 "日志目录不存在"

**解决方案**：
```bash
# 创建日志目录
mkdir -p logs

# 检查日志配置
# 确保 server/logger.ts 配置正确
```

## 配置优化

### 推荐配置

| 场景 | interval | timeout | auto-approve | max-retries | 说明 |
|------|----------|---------|--------------|-------------|------|
| 快速开发 | 20 | 60 | ❌ | 0 | 快速检测，适合短任务 |
| 标准开发 | 30 | 120 | ❌ | 0 | 默认配置，平衡性能和响应 |
| 长时间任务 | 60 | 300 | ❌ | 0 | 减少误报，适合大型 Spec |
| 调试模式 | 10 | 30 | ❌ | 0 | 快速反馈，便于调试 |
| 无人值守 | 30 | 120 | ✅ | 3-5 | 自动恢复 + 限制重试 |
| CI/CD | 60 | 300 | ✅ | 3 | 长超时 + 自动化 + 防止死循环 |
| 网络不稳定 | 30 | 180 | ✅ | 10 | 更长超时 + 更多重试 |

### 性能影响

- CPU 占用：< 1%（空闲时）
- 内存占用：< 50MB
- 磁盘 I/O：每次检查读取日志文件末尾（< 10KB）

5. **生产环境不需要监控**
   - 监控脚本仅用于开发环境
   - 生产环境使用其他监控方案（如 PM2、Docker healthcheck）

6. **定期清理恢复标记文件**
   ```bash
   # 清理所有恢复标记
   rm .kiro-resume-*
   ```

7. **自动同意模式的安全建议**
   - ✅ 适用：数据录入、测试运行、文档生成等安全操作
   - ⚠️ 谨慎：涉及删除、重构、部署等高风险操作
   - ❌ 禁止：生产环境操作、未经审查的代码变更

8. **重试次数设置建议**
   - **无限制（0）**：适合稳定网络环境，任务可靠
   - **3-5 次**：适合一般场景，平衡自动化和安全性
   - **10+ 次**：适合网络不稳定环境
   - **1 次**：适合高风险操作，只允许一次自动恢复

9. **智能计数重置**
   - 任务正常完成后（持续活动 5 分钟），计数自动重置为 0
   - 无需手动清理计数文件
   - 新任务从 0 开始计数

## 重试机制详解

### 工作原理

```
任务开始
   ↓
执行中...
   ↓
网络中断 → 监控检测到 → 创建恢复标记（计数: 1）
   ↓
Hook 触发 → Kiro 恢复 → 继续执行
   ↓
再次中断 → 监控检测到 → 创建恢复标记（计数: 2）
   ↓
Hook 触发 → Kiro 恢复 → 继续执行
   ↓
再次中断 → 监控检测到 → 创建恢复标记（计数: 3）
   ↓
Hook 触发 → Kiro 恢复 → 继续执行
   ↓
再次中断 → 监控检测到 → 已达最大重试次数 → 停止自动恢复
   ↓
等待人工处理
```

### 计数重置机制

脚本会自动检测任务是否正常完成：

- **检测条件**：距离上次恢复超过 5 分钟，且日志文件持续有活动
- **重置操作**：将恢复计数重置为 0
- **好处**：新任务从 0 开始计数，无需手动清理

### 计数文件

- **位置**：`.kiro-resume-count-<sessionId>`
- **内容**：
  ```json
  {
    "count": 3,
    "lastResumeTime": 1709481234567,
    "sessionId": "2026-03-03"
  }
  ```
- **清理**：任务正常完成后自动删除，或手动删除 `rm .kiro-resume-count-*`

### 示例输出

```bash
# 第 1 次中断
⚠️  [2026-03-03] 检测到执行中断（无活动 125秒）
🔢 [2026-03-03] 恢复次数: 1/3
🔄 [2026-03-03] 已创建恢复标记: .kiro-resume-2026-03-03
📊 [2026-03-03] 剩余恢复次数: 2

# 第 2 次中断
⚠️  [2026-03-03] 检测到执行中断（无活动 130秒）
🔢 [2026-03-03] 恢复次数: 2/3
🔄 [2026-03-03] 已创建恢复标记: .kiro-resume-2026-03-03
📊 [2026-03-03] 剩余恢复次数: 1

# 第 3 次中断
⚠️  [2026-03-03] 检测到执行中断（无活动 128秒）
🔢 [2026-03-03] 恢复次数: 3/3
🔄 [2026-03-03] 已创建恢复标记: .kiro-resume-2026-03-03
📊 [2026-03-03] 剩余恢复次数: 0

# 第 4 次中断（达到上限）
⚠️  [2026-03-03] 检测到执行中断（无活动 132秒）
⛔ [2026-03-03] 已达到最大恢复次数 (3)，停止自动恢复
```

## 限制和注意事项

1. **仅监控日志文件**
   - 脚本通过日志文件判断活动状态
   - 如果日志系统失效，监控也会失效

2. **无法处理所有中断场景**
   - 网络中断：✅ 可以检测
   - 进程崩溃：❌ 无法检测（需要进程管理器）
   - 系统重启：❌ 无法检测（需要系统级监控）

3. **Hook 触发延迟**
   - 检查间隔 + Hook 响应时间
   - 默认配置下，最长延迟约 30-60 秒

4. **多窗口识别依赖日志文件名**
   - 确保不同窗口使用不同的日志文件
   - 或者使用会话 ID 区分

5. **重试次数限制**
   - 设置 `--max-retries=0` 表示无限制（默认）
   - 达到上限后停止自动恢复，需要人工介入
   - 计数文件存储在 `.kiro-resume-count-*`

6. **计数重置条件**
   - 任务正常完成后（持续活动 5 分钟）自动重置
   - 或手动删除计数文件：`rm .kiro-resume-count-*`
